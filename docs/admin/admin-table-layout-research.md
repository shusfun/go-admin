# 宽表竞品调研 v2

更新日期：2026-04-10

## 1. 这次怎么调研

上一版偏“表格能力清单”，不够像真正的竞品拆解。

这次换成更直接的方法：

- 只看真实产品里“字段很多”的列表界面
- 重点看它们怎么组织页面骨架、表头、主列、次级信息、详情区
- 少谈抽象能力，多谈具体渲染方式

本次主要参考：

- Airtable
- Notion
- Jira
- Shopify Admin
- Asana
- monday.com
- ClickUp

说明：

- 下文里“值得借鉴的做法”是基于官方帮助中心/官方文档和产品公开说明做的工程归纳
- 重点不是抄视觉，而是抄它们如何降低宽表阅读成本

---

## 2. 先给结论

真正优秀的宽表，核心都不是“把更多字段塞进表里”，而是三件事：

1. 把列表当成工作台，不把它当数据库导出页
2. 让主信息固定可读，让次信息延后出现
3. 用视图分流需求，而不是让所有角色共用一张超宽默认表

换句话说，很多产品并不是“宽表渲染特别牛”，而是：

- 默认只展示任务完成所需字段
- 其他字段通过视图、展开、侧栏、详情页再看

如果一张表默认要同时满足“运营、财务、客服、风控、审计”五类人，它一定会越来越宽。

---

## 3. 竞品到底怎么做

## 3.1 Airtable：宽表不是问题，前提是每个人都在自己的视图里看

从 Airtable 的能力能看出几个明显倾向：

- 视图是核心对象，不是附属功能
- 字段可以隐藏、排序、分组、冻结
- 主字段永远承担“识别这行是谁”的责任
- 用户并不被鼓励把所有字段同时打开

最值得抄的不是它“像 Excel”，而是这套字段治理思路：

- 第一列一定是身份字段
- 冻结字段数量非常克制
- 低频字段默认隐藏
- 同一张表允许不同人有不同视图

它的渲染方式本质是：

```text
┌──────────────────────────────────────────────────────────────────────┐
│ 视图 tabs   筛选   分组   排序   隐藏字段   冻结字段                 │
├──────────────┬───────────────────────────────────────────────────────┤
│ 主字段       │ 当前视图真正关心的列                                  │
│ 客户名称     │ 状态 | 负责人 | 最新跟进 | 金额 | 来源 | 标签 ...      │
├──────────────┼───────────────────────────────────────────────────────┤
│ XX 公司      │ 已签约 | 王五 | 今天 | 30w | 转介绍 | A 类 ...        │
│ YY 公司      │ 跟进中 | 李四 | 昨天 | 12w | 线索池 | B 类 ...        │
└──────────────┴───────────────────────────────────────────────────────┘
```

关键信号：

- Airtable 不是解决“怎么一次看完所有字段”
- Airtable 是解决“怎么在当前视图里只看有用字段”

这点很重要。

## 3.2 Notion：表格只是数据库的一个视图，不是最终信息承载层

Notion 的思路比 Airtable 更激进：

- 表格只承载摘要
- 完整信息大多在记录详情页里
- 属性显示与隐藏本来就是视图配置的一部分

这意味着：

- Notion 并不追求让表格本身成为完整阅读终点
- 表格更像“索引页”

它适合字段很多但首屏不该拥挤的场景，典型渲染逻辑是：

```text
┌──────────────────────────────────────────────────────────────┐
│ 视图 tabs   Filter   Sort   Group   Properties              │
├──────────────┬───────────────────────────────────────────────┤
│ 标题列       │ 状态 | 负责人 | 截止时间 | 优先级 | 标签      │
├──────────────┼───────────────────────────────────────────────┤
│ 新店开业计划 │ 进行中 | 张三 | 04-20 | P1 | 华东            │
│ 供应商复核   │ 待处理 | 李四 | 04-18 | P2 | 采购            │
└──────────────┴───────────────────────────────────────────────┘

点击行后:

┌──────────────────────────────────────────────────────────────┐
│ 记录详情页 / 侧栏                                             │
│ 描述、备注、附件、关联记录、历史、长文本、富文本、子任务 ...   │
└──────────────────────────────────────────────────────────────┘
```

最值得借鉴的点：

- 宽表里最不该出现的往往是长文本、说明、备注、上下文
- 这些内容应该转移到详情面板，而不是继续加列

## 3.3 Jira：问题很多、字段也很多，但默认列表仍然只围绕“排查/筛选/批处理”

Jira 的 Issue Navigator 很典型：

- 顶部是过滤器和搜索
- 中间是问题列表
- 列是可配置的
- 真正宽的字段体系在“issue 详情”里，而不是默认列表里

它的表格渲染并不追求信息铺满，而是围绕“快速筛选后批量处理”：

```text
┌──────────────────────────────────────────────────────────────────────┐
│ Saved filter / Search / Assignee / Status / More                    │
├──────────────────────────────────────────────────────────────────────┤
│ Key | Summary | Status | Assignee | Priority | Updated              │
├──────────────────────────────────────────────────────────────────────┤
│ OPS-231 | 修复结算异常   | In QA  | Alice    | High     | 2h ago    │
│ OPS-228 | 新增导出字段   | To Do  | Bob      | Medium   | 1d ago    │
└──────────────────────────────────────────────────────────────────────┘
```

Jira 给的启发是：

- 主列表要服务于“找、筛、改、批量操作”
- 不是服务于“展示系统所有字段”

对后台业务表尤其重要。

## 3.4 Shopify Admin：同一份数据，通过视图把岗位需求拆开

Shopify 的订单、商品、客户列表都很像一件事：

- 顶部一定先是 saved views
- 然后才是 filter / sort / column
- 默认每个视图只保留和当前任务强相关的列

它不是先给你一张大表，再让你自己慢慢删列；而是先给你任务导向的视图。

典型骨架：

```text
┌──────────────────────────────────────────────────────────────────────┐
│ 视图: 全部订单 | 待发货 | 已退款 | 高风险 | 我的视图                │
├──────────────────────────────────────────────────────────────────────┤
│ Filter   Sort   Columns   Export                                    │
├──────────────────────────────────────────────────────────────────────┤
│ 订单号 | 客户 | 支付状态 | 发货状态 | 渠道 | 金额 | 下单时间         │
├──────────────────────────────────────────────────────────────────────┤
│ #1001  | 张三 | 已支付   | 待发货   | 小程序 | 129   | 今天 10:32    │
└──────────────────────────────────────────────────────────────────────┘
```

值得学的点：

- “字段很多”先不要想技术怎么渲染
- 先问是不是你把多个工作台需求揉在了一张表里

## 3.5 Asana / ClickUp：列表只承载短字段，重字段进右侧详情

这类任务管理产品的共性很稳定：

- 行内展示少量可快速扫读的字段
- 标题列承担最多信息量
- 备注、描述、评论、附件、子任务不放进表格
- 点击行后，右侧详情面板承接重内容

也就是说，它们对单元格的要求不是“什么都能塞”，而是：

- 列表只负责扫读
- 详情才负责深读

ASCII：

```text
┌──────────────────────────────────────────────────────────────┬────────┐
│ 任务名                | 状态 | 负责人 | 截止时间 | 项目       │ 详情栏 │
├──────────────────────────────────────────────────────────────┼────────┤
│ 门店巡检 SOP 更新      | 进行中 | 王五  | 04-16    | 运营标准化 │ 标题   │
│ 供应商合同补录         | 待办   | 李四  | 04-18    | 采购       │ 描述   │
│                                                              │ 评论   │
│                                                              │ 附件   │
│                                                              │ 历史   │
└──────────────────────────────────────────────────────────────┴────────┘
```

这类布局对“日志、工单、审批、任务”非常有效。

## 3.6 monday.com：字段多时，不强行平铺，而是借助分组和板块

monday 的表格常见两个手法：

- 先按组分块
- 再在组内做表格

这样用户不是面对一整张无限延展的宽表，而是面对“多个有语义的小表段”。

ASCII：

```text
项目组 A
┌──────────────────────────────────────────────────────────────┐
│ 名称 | owner | 状态 | timeline | budget | risk | next step   │
├──────────────────────────────────────────────────────────────┤
│ ...                                                          │
└──────────────────────────────────────────────────────────────┘

项目组 B
┌──────────────────────────────────────────────────────────────┐
│ 名称 | owner | 状态 | timeline | budget | risk | next step   │
├──────────────────────────────────────────────────────────────┤
│ ...                                                          │
└──────────────────────────────────────────────────────────────┘
```

这个思路适合天然分组明显的业务，比如：

- 按门店
- 按区域
- 按项目阶段
- 按履约状态

## 4. 真正值得抄的，不是样子，而是这 8 个渲染动作

## 4.1 顶部先有视图 tabs，再有列设置

优秀产品很少把“列设置”放成唯一入口。

更常见的是：

- 先选工作视图
- 再微调当前视图里的列

原因很简单：

- 视图解决 80% 的“该看什么”
- 列设置解决剩下 20% 的个性化

这比一上来让用户面对 30 个复选框好很多。

## 4.2 第一列不是普通列，而是身份列

大多数成熟产品都在强化第一列：

- 订单号
- 任务标题
- 客户名
- Issue Key + Summary

它不是简单显示一个字段，而是承载“这一行到底是谁”。

更好的做法通常是：

- 主标题一行
- 次信息一行

例如：

```text
┌──────────────────────┐
│ 客户名称 / 订单标题   │
│ 编号 / 次级标签 / 来源│
└──────────────────────┘
```

这样可以少开 1 到 3 列。

## 4.3 状态字段尽量压缩到徽标、颜色、短标签

优秀列表很少把状态写成长句。

常见压缩方式：

- 标签
- 徽标
- 小图标 + 文本
- 颜色块

如果一个状态列占到 14 到 20 个字，宽表一定会被拖坏。

## 4.4 备注、说明、URL、JSON、错误详情不要继续占列

这类字段最容易把表拉宽，但对扫读价值最低。

成熟产品通常这样处理：

- 截断
- hover
- 展开行
- 右侧详情栏
- 单独详情页

如果一个字段需要换行阅读，它大概率就不属于主表常驻列。

## 4.5 操作列不求丰富，只求稳定

优秀产品的操作列通常很克制：

- 最多保留 1 到 2 个高频动作
- 其他动作收进更多菜单

否则右侧固定操作列会越来越肥，最后吞掉中间有效内容区。

## 4.6 宽表不是“全局页面横向滚动”，而是“表区域自己滚”

这是很多后台最容易犯的错。

成熟产品基本都是：

- 顶部筛选区不跟着横滚
- 页头不跟着横滚
- 只有表体区域横滚
- 主列/操作列必要时固定

也就是说，滚动应该被局部化。

## 4.7 真正高密度的场景，会合并单元格信息，而不是继续加列

例如把：

- 姓名
- 编号
- 手机尾号

合并到一格里；

把：

- 支付状态
- 发货状态

做成双 badge 叠放到同一格里。

这样比“再多开两列”稳定得多。

ASCII：

```text
┌────────────────────────┬──────────────────┐
│ 客户                    │ 履约             │
├────────────────────────┼──────────────────┤
│ 张三                    │ [已支付] [待发货]│
│ U1023 / 138****8899     │                  │
└────────────────────────┴──────────────────┘
```

## 4.8 当列真的很多时，先砍默认展示列，再谈虚拟滚动

虚拟滚动只能解决性能，不能解决可读性。

正确顺序应该是：

1. 先做视图
2. 再做列分级
3. 再做详情外移
4. 最后才是虚拟化

## 5. 哪些做法其实不值得抄

## 5.1 默认展示所有字段

这通常只适合内部导出页，不适合日常工作台。

## 5.2 一有宽表就上双层表头

双层表头只适合“字段分组天然稳定”的报表。

对普通 CRUD，双层表头很容易显得沉重。

## 5.3 冻结太多列

冻结列一多，中间滚动区马上失去价值。

从竞品习惯看，真正长期好用的通常只冻结：

- 1 个主身份列
- 或者再加 1 个关键状态 / 选择列

## 5.4 在表内塞完整业务上下文

如果你开始想把：

- 审批备注
- 接口报错
- JSON 参数
- 操作历史

都放进表里，说明这张表已经不该只是表了。

## 6. 对当前后台更像样的 3 套方案

这次不再给“泛泛的四选一”，而是给三套更接近产品形态的骨架。

## 6.1 方案 A：Jira + Shopify 型工作台

适合：

- 标准业务列表
- 订单、用户、商品、门店、供应商

页面骨架：

```text
┌──────────────────────────────────────────────────────────────────────────┐
│ PageHeader: 标题 / 主操作                                               │
├──────────────────────────────────────────────────────────────────────────┤
│ ViewTabs: 全部 | 待处理 | 异常 | 我的视图                               │
├──────────────────────────────────────────────────────────────────────────┤
│ FilterBar: 搜索 | 状态 | 时间 | 渠道 | 更多筛选 | 列设置                 │
├───────────────┬──────────────────────────────────────────┬───────────────┤
│ 身份列 sticky │ 中间业务字段可横滚                        │ 操作列 sticky │
│ 名称 + 编号   │ owner | status | tag | amount | source   │ 状态/更多     │
├───────────────┼──────────────────────────────────────────┼───────────────┤
│ 两行信息压缩  │ 中字段只留决策信息                        │ 高频动作收口  │
└───────────────┴──────────────────────────────────────────┴───────────────┘
```

这套最适合你们常规 CRUD。

## 6.2 方案 B：Notion + ClickUp 型“列表 + 详情栏”

适合：

- 日志
- 工单
- 审批
- 任务
- 任何长文本/上下文很多的记录

页面骨架：

```text
┌───────────────────────────────────────────────────────────────┬──────────┐
│ ViewTabs / FilterBar                                          │ 详情栏   │
├───────────────────────────────────────────────────────────────┼──────────┤
│ 标题列 | 状态 | owner | 时间 | 来源 | 操作                    │ 标题     │
│ 行内只保留摘要                                                 │ 备注     │
│                                                               │ 参数     │
│                                                               │ 历史     │
│                                                               │ 附件     │
└───────────────────────────────────────────────────────────────┴──────────┘
```

这套非常适合把“宽度压力”转成“深度阅读”。

## 6.3 方案 C：Airtable + monday 型“分组宽表”

适合：

- 报表
- 履约看板
- 多指标业务盘点

页面骨架：

```text
┌──────────────────────────────────────────────────────────────────────┐
│ ViewTabs / GroupBy / Sort / HiddenFields / Density                  │
├──────────────────────────────────────────────────────────────────────┤
│ Group: 华东                                                         │
│ ┌────────────┬────────────────────────────────────────────────────┐ │
│ │ sticky 主列 │ 指标列可横滚                                     │ │
│ └────────────┴────────────────────────────────────────────────────┘ │
│ Group: 华南                                                         │
│ ┌────────────┬────────────────────────────────────────────────────┐ │
│ │ sticky 主列 │ 指标列可横滚                                     │ │
│ └────────────┴────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────────────┘
```

它不适合所有页面，但对“按区域/阶段分块”的场景很有帮助。

## 7. 我现在更明确的建议

如果目标是把你们后台的列表做得像成熟产品，而不是像“把数据库字段排成表”，我会建议这样落：

### 7.1 标准 CRUD 页

直接抄 `Jira + Shopify`：

- 上面先有 `ViewTabs`
- 下面是 `FilterBar`
- 表内只保留身份列、关键业务列、操作列
- 身份列做成两行复合单元格
- 低频列放到列设置

### 7.2 日志/记录页

直接抄 `Notion + ClickUp`：

- 主表只看摘要
- 点击行开右侧详情
- 请求参数、错误详情、原始响应、备注都去详情栏

### 7.3 报表/盘点页

直接抄 `Airtable + monday`：

- 先按业务分组
- 再在组内横向比指标
- 不追求一张巨表平铺到底

---

## 8. 对当前仓库最重要的翻译

你们现有 `admin-web` 多数页面还是原生 `table`，这并不是问题。

真正的问题是：

- 还缺“视图层”
- 还缺“身份列复合渲染”
- 还缺“列表摘要化 + 详情外移”

所以如果后续真的要改，我认为优先级应该是：

1. 给列表加 `ViewTabs`
2. 给第一列做复合渲染，减少 1 到 3 个散列
3. 给日志/记录页做右侧详情
4. 再做列显隐
5. 最后才考虑更重的表格状态层
